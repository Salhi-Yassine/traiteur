# Takeover Audit — 2026-07-23

> Project handover review. Findings from a full pass over CI, backend, frontend,
> tests, and configuration, ordered by severity. Items marked ✅ were fixed the
> same day (PRs #44, #45).

---

## P0 — Broken foundations (blockers for trusting anything else)

### 1. ✅ CI red on every run since early May (fixed in PR #45)
Two independent blockers meant no pipeline had passed for months:
- `pwa/Dockerfile` activated `pnpm@latest`; pnpm 10 broke `pnpm config -g set`
  in `node:lts`. → pinned to `pnpm@9.1.3` (matches `packageManager`).
- The migration chain could not replay on a fresh database — `make full-migrat`'s
  cleanup step pruned early migrations, leaving later ones referencing tables
  that never get created. CI's `migrate` + `schema:validate` steps could never
  pass. → squashed to a single baseline migration generated from entity metadata.
**Follow-up:** the `full-migrat` Makefile target must stop deleting old
migrations, or every squash will rot again. E2E workflow shares the Docker fix
but has not been re-verified end-to-end.

### 2. ✅ Backend test suite entirely red (fixed in PR #44)
- Test database schema had drifted (`google_id` missing) and could not be
  migrated forward.
- `AuthTest` used pre-prefix routes (`/users`, `/me`) and an invalid
  `userType: 'client'` — it can never have passed as written.
- PHPStan had 15 unbaselined false positives → `make cs` red on main.
- 3 ESLint errors on main → `make pnpm c="lint"` red.

### 3. API response shape mismatch — features silently running on mock data
API Platform 4 returns collections under `member`, but most pages read
`hydra:member` (checklist, invites, budget, dashboard, magazine, vendors/index
SSR…). Where the read fails, pages fall back to embedded mock data — so
several features marked "done" have likely never displayed real API data.
**Decision needed:** either set `api_platform.serializer.hydra_prefix: true`
(one-line backend fix, least churn) or migrate the frontend to `member` via a
single shared `unwrapCollection()` helper (forward-compatible; AP5 drops the
prefix). Recommendation: shared helper + `member`, done in one sweep with tests.

### 4. TypeScript is not enforced anywhere — 21 real errors on main
`tsc --noEmit` fails with 21 errors (budget.tsx implicit `any`s, iterator
target issues, wrong prop types in BudgetDonutChart, InspirationGrid…).
Neither the DoD, CI, nor `next build`'s output is gating on it.
**Action:** fix the 21 errors, add `"typecheck": "tsc --noEmit"` script, add it
to CI and to CLAUDE.md's DoD.

## P1 — Correctness & production hygiene

### 5. Mock/demo data and a hardcoded clock inside product pages
`pages/mariage/budget.tsx` ships `MOCK_BUDGET_ITEMS`, `SEED_BUDGET`, a
hardcoded `WEDDING_DATE` and — worst — `const TODAY = new Date("2026-04-22")`.
Overdue/payment logic is frozen in time. Similar mock fallbacks exist in
magazine, dashboard, invites, checklist ("demo mode"). Real data wiring depends
on fixing P0-3 first. Demo mode should be an explicit, visible state, not a
silent fallback.

### 6. Committed secrets / weak defaults
- `api/.env` commits a real-looking `JWT_PASSPHRASE`.
- `APP_SECRET=` is empty; Postgres default password `!ChangeMe!` in compose.
Before prod: rotate JWT keypair + passphrase, generate APP_SECRET, move all
secrets to environment/secret manager; document in `.env.example` only.

### 7. Backend test coverage far below the project's own DoD
Only Auth/AppStats/SavedVendor have API tests. Security-critical surfaces with
zero coverage: public RSVP token endpoint (`/api/public/guests/{token}`),
QuoteRequest visibility, WeddingProfile voters, vendor profile edit ownership.
One `ApiTestCase` per entity covering 200/201/403/404 is the standing rule —
backfill starting with the security-sensitive ones.

### 8. DoD violations in shipped UI
- 3 × `alert()` (e.g. vendor profile share fallback) — rule says `toast.*`.
- Hardcoded French strings in `pages/magazine/index.tsx` ("Vrais Mariages",
  "Ils ont dit oui sur Farah", "Voir tous →") and its mock arrays.
- ~20 `: any` in non-test frontend files despite `no-explicit-any`.

### 9. Project tracking is stale and misleading
- Issues #11, #12, #14, #17, #18, #19, #21, #28 (half), #29, #31 describe work
  that is already shipped — close them after code-verification.
- `PROGRESS.md` (2026-04-17, "~38%") contradicts CLAUDE.md (2026-05-04, "~48%").
  Pick one source of truth (suggest: CLAUDE.md status table + DONE.md log;
  retire PROGRESS.md or regenerate it).

## Standing decisions taken during this audit
- Test databases are rebuilt from entity metadata (`doctrine:schema:create` or
  the squashed baseline); the dev DB keeps its data with the baseline registered
  as executed.
- Migration squash: from now on, keep migrations replayable — stop pruning.
